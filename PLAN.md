# Plan: Build a Bun CLI for adversarial implement→verify loops on top of pi

## Goal

Build a standalone **Bun CLI** that:

1. takes an explicit plan file
2. validates repo + tooling preconditions
3. creates a feature branch from a selected base branch
4. runs a fresh-context **implementer** step
5. runs a fresh-context **verify** step
6. filters findings by severity threshold
7. loops until either:
   - no findings remain with severity `>= threshold`, or
   - max turns is reached
8. commits after each turn with repo changes
9. pushes once at the end
10. creates a **draft PR/MR** via `gh` or `glab`
11. stores rich artifacts under `~/.local/state/adversary/<repo>-<hash>/runs/...`

---

# Locked decisions

## Runtime / shape
- Standalone CLI
- Built with **Bun**
- One process per repo / working directory
- One run = one plan = one branch = one PR/MR
- Fresh context window each implement/verify step via external process invocation

## Repo / git behavior
- Must be inside a git repo
- Fail if repo is not git
- Require **completely clean** startup state:
  - no staged changes
  - no unstaged changes
  - no untracked files
- Resolve base branch via:
  1. CLI override
  2. config override
  3. autodetect
- **No auto-sync/fetch**
- Tool checks out base branch automatically
- Tool creates feature branch immediately after validation
- Start from selected base branch, not current arbitrary HEAD
- Leave repo on feature branch at end
- Commit after every turn
- Push only once at end
- Local branch name collision: auto-suffix
- Remote branch collision on push: fail

## Loop semantics
- `turns >= 1`
- no hard max ceiling
- severity threshold is numeric only
- threshold valid range is **1..10**
- `verify` severity semantics:
  - higher number = more severe
  - fix findings with severity `>= threshold`
- stop when latest verify report contains **zero** findings `>= threshold`
- below-threshold findings do **not** affect loop control
- below-threshold findings are still preserved and shown in summaries/PR body

## Step failure behavior
- step failure = non-zero exit code or missing/invalid required artifact
- implement failure: stop run
- verify failure: stop run
- verify `blocked`: treat as terminal like error
- PR/MR creation failure: non-zero process exit
- overall process exits 0 if workflow completes end-to-end, even if threshold findings remain
- non-zero only for operational failures

## Commands / prompting
- Spawn external commands
- Default implementer command: external **`pi -p`**
- Default verify command: external **`pi -p`**
- Minimal defaults: inherit normal pi config/environment
- Config supports command **templates with variables**
- First-turn prompt:
  - plan
  - threshold
  - instructions to implement proactively for likely verification success
  - tell agent not to manage git
  - tell agent not to run verify
- Later-turn prompt:
  - original plan
  - current filtered findings `>= threshold`
  - compact deterministic run history
  - exact threshold
  - instruction to focus only on current threshold findings + plan

## Artifacts
- Fixed artifact root: `~/.local/state/adversary/<repo>-<hash>/runs/<timestamp>-<plan-slug>/`
- keep all runs forever in v1
- full-fidelity artifacts, no redaction
- plan may live anywhere on disk
- snapshot plan into run dir at startup
- use snapshotted copy as canonical run input
- plan must be readable and non-empty
- any text file accepted

## Summaries
- final summary in both:
  - `final-summary.md`
  - `final-summary.json`
- each turn also gets machine-readable summary JSON
- final summary includes:
  - high-level result
  - threshold findings in full
  - below-threshold findings in full too
- raw stdout/stderr stay in artifacts, not PR body

## PR/MR behavior
- always create PR/MR at end
- always create as **draft/WIP**
- use `gh` for GitHub, `glab` for GitLab
- deterministic PR/MR body generated by orchestrator
- PR title source:
  - first heading in plan
  - fallback to filename

## Safety / preflight
- preflight command existence:
  - git
  - pi
  - gh/glab as applicable
- preflight auth/readiness for gh/glab
- preflight verify structured-output contract support

# Deliverables

## Deliverable 1: extend `verify` with machine-readable output
This is a dependency for the orchestrator.

### Required changes
Add structured output support to `verify`, minimally:

- support explicit arguments like:
  - `--format=json`
  - `--output=/path/to/file.json`

### Required top-level JSON fields
Minimal v1 contract:

```json
{
  "schemaVersion": 1,
  "status": "ok|blocked|error",
  "findings": [
    {
      "title": "string",
      "severity": 7,
      "location": {
        "path": "src/file.ts",
        "line": 42,
        "column": 1
      },
      "description": "string",
      "sources": ["reviewer", "qa"]
    }
  ]
}
```

### Notes
- findings include **all** findings, not filtered by threshold
- no stable finding IDs required in v1
- small status enum only
- orchestrator filters findings itself

### Acceptance criteria
- verify can emit valid JSON to requested output path
- JSON schema is stable enough for the orchestrator
- command exits non-zero or reports `status != ok` appropriately on failure/blockage

# Implementation phases

## Phase 1: define config and artifact schema
Create the internal schemas first.

### Config schema
Repo-local config file, overridden by CLI flags.

Suggested fields:

```json
{
  "baseBranch": "main",
  "implementCommandTemplate": "pi -p @{promptFile}",
  "verifyCommandTemplate": "pi -p \"/skill:verify --mode=report-only --format=json --output={verifyOutputFile}\"",
  "implementTimeoutMs": 10800000,
  "verifyTimeoutMs": 5400000,
  "prTimeoutMs": 300000

```

## Phase 6: verify parsing and loop control
Implement the core adversarial loop.

### Logic
For each turn:
1. generate implement prompt files
2. run implement command
3. detect repo changes
4. if repo changes exist:
   - commit with per-turn message
5. run verify
6. parse verify JSON
7. split findings:
   - threshold findings `>= threshold`
   - below-threshold findings
8. if zero threshold findings:
   - mark clean success
   - stop
9. else if turns remain:
   - continue
10. else:
   - mark capped with remaining findings

### No-op turn rule
- no repo changes after implement is allowed
- skip commit
- still run verify

### Acceptance criteria
- loop stops correctly
- threshold logic correct
- current-report-only unresolved logic used
